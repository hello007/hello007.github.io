1. LangChain4j

🎯 掌握程度：必须掌握

什么是LangChain4j？

通俗解释：
LangChain4j是"AI应用的Spring Boot"。就像Spring Boot简化了Web开发，LangChain4j简化了AI应用开发。

技术定义：
Java语言的AI应用开发框架，提供LLM调用、RAG、Agent、Prompt管理等完整功能。

为什么选择LangChain4j？

LangChain4j的优势：
✅ 原生Java：纯Java实现，不是Python移植
✅ 功能完善：覆盖RAG、Agent、Memory等
✅ 易于集成：支持Spring Boot、Quarkus
✅ 性能优秀：针对JVM优化
✅ 社区活跃：快速迭代，文档完善

@Service

            SearchRequest.query(question).withTopK(5)
        );

        // 2. 构建上下文
        String context = documents.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n\n"));

        // 3. 生成回答
        String prompt = String.format("""
            基于以下上下文回答问题：

            上下文：
            %s

            问题：%s
            """, context, question);

        return chatModel.call(prompt);
    }
}

3. API Rate Limiting（速率限制）

🎯 掌握程度：必须掌握

什么是速率限制？

通俗解释：
就像高速公路收费站限制车辆通行速度，API速率限制保护服务不被过度调用。

技术定义：
控制API请求频率的机制，防止服务过载和控制成本。

为什么需要速率限制？

速率限制的作用：
✅ 成本控制：防止API调用费用失控
✅ 服务保护：避免超过提供商限制
✅ 性能优化：合理分配系统资源
✅ 用户体验：确保响应时间稳定
✅ 合规要求：遵守API使用条款

常见限制类型

限制类型 说明 常见值
TPM (Tokens Per Minute) 每分钟Token数 90,000 (GPT-3.5)
RPM (Requests Per Minute) 每分钟请求数 3,000 (GPT-3.5)
TPD (Tokens Per Day) 每天Token数 150,000 (免费版)
Concurrent 并发请求数 50-100

实现策略

1) 令牌桶算法

@Component
public class RateLimiter {

    private final Semaphore semaphore;
    private final ScheduledExecutorService scheduler;

    public RateLimiter(int maxRequests, int periodSeconds) {
        this.semaphore = new Semaphore(maxRequests);
        this.scheduler = Executors.newScheduledThreadPool(1);

        // 定期补充令牌
        scheduler.scheduleAtFixedRate(() -> {
            semaphore.release(maxRequests - semaphore.availablePermits());
        }, 0, periodSeconds, TimeUnit.SECONDS);
    }

    public boolean tryAcquire() {
        return semaphore.tryAcquire();
    }

    public void acquire() throws InterruptedException {
        semaphore.acquire();
    }
}

2) 滑动窗口算法

@Component
public class SlidingWindowRateLimiter {

    private final ConcurrentLinkedDeque<Long> timestamps;
    private final int maxRequests;
    private final long windowSizeMillis;

    public SlidingWindowRateLimiter(int maxRequests, int windowSizeSeconds) {
        this.maxRequests = maxRequests;
        this.windowSizeMillis = windowSizeSeconds * 1000L;
        this.timestamps = new ConcurrentLinkedDeque<>();
    }

    public synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();

        // 移除窗口外的时间戳
        while (!timestamps.isEmpty() &&
               now - timestamps.peekFirst() > windowSizeMillis) {
            timestamps.removeFirst();
        }

        // 检查是否允许请求
        if (timestamps.size() < maxRequests) {
            timestamps.addLast(now);
            return true;
        }

        return false;
    }
}

3) Redis分布式限流

@Service
public class RedisRateLimiter {

    @Autowired
    private StringRedisTemplate redisTemplate;

    public boolean allowRequest(String key, int limit, int windowSeconds) {
        long now = System.currentTimeMillis();
        long windowStart = now - (windowSeconds * 1000);

        String keyWithTime = "rate_limit:" + key;

        // 使用Lua脚本保证原子性
        String luaScript = """
            local key = KEYS[1]
            local now = tonumber(ARGV[1])
            local window = tonumber(ARGV[2])
            local limit = tonumber(ARGV[3])

            -- 清理过期记录
            redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

            -- 计数
            local count = redis.call('ZCARD', key)

            if count < limit then
                redis.call('ZADD', key, now, now)
                redis.call('EXPIRE', key, window / 1000)
                return 1
            else
                return 0
            end
            """;

        DefaultRedisScript<Long> script = new DefaultRedisScript<>(luaScript, Long.class);
        Long result = redisTemplate.execute(
            script,
            Collections.singletonList(keyWithTime),
            String.valueOf(now),
            String.valueOf(windowSeconds * 1000),
            String.valueOf(limit)
        );

        return result == 1;
    }
}

@Configuration
public class RateLimitConfig {

    @Bean
    public FilterRegistrationBean<RateLimitFilter> rateLimitFilter() {
        FilterRegistrationBean<RateLimitFilter> registration = new FilterRegistrationBean<>();
        registration.setFilter(new RateLimitFilter());
        registration.addUrlPatterns("/api/*");
        registration.setOrder(1);
        return registration;
    }
}

@Component
public class RateLimitFilter implements Filter {

    @Autowired
    private RateLimiter rateLimiter;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                        FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        HttpServletResponse res = (HttpServletResponse) response;

        String key = getClientIdentifier(req);

        if (!rateLimiter.allowRequest(key)) {
            res.setStatus(429); // Too Many Requests
            res.getWriter().write("{\"error\":\"Rate limit exceeded\"}");
            return;
        }

        chain.doFilter(request, response);
    }

    private String getClientIdentifier(HttpServletRequest request) {
        // 使用IP或用户ID作为标识
        return request.getRemoteAddr();
    }
}

4. Caching Strategy（缓存策略）

🎯 掌握程度：必须掌握

什么是缓存策略？

通俗解释：
就像记笔记避免重复思考，AI缓存存储常见问题的答案，避免重复调用API。

技术定义：
存储AI响应结果，相同或相似请求直接返回缓存，减少API调用和成本。

为什么需要缓存？

缓存的好处：
✅ 降低成本：减少API调用费用
✅ 提升性能：缓存响应快于API调用
✅ 减少延迟：避免网络请求时间
✅ 减轻负载：保护API服务
✅ 提高可用性：API故障时仍可响应

缓存策略类型

策略类型 适用场景 实现复杂度 效果
精确匹配 FAQ、固定问题 低 高
语义相似 相似问题重述 高 中高
局部缓存 会话上下文 中 中
分层缓存 高并发场景 高 高

2) 语义相似缓存

@Service
public class SemanticCacheService {

    @Autowired
    private EmbeddingModel embeddingModel;

    @Autowired
    private VectorStore vectorStore;

    private static final double SIMILARITY_THRESHOLD = 0.95;

    public String getCachedAnswer(String question) {
        // 1. 生成问题向量
        Embedding queryEmbedding = embeddingModel.embed(question).content();

        // 2. 搜索相似缓存
        List<EmbeddingMatch> matches = vectorStore.findRelevant(
            queryEmbedding,
            5
        );

        // 3. 检查是否足够相似
        for (EmbeddingMatch match : matches) {
            if (match.score() >= SIMILARITY_THRESHOLD) {
                CachedAnswer cached = (CachedAnswer) match.embedded();
                return cached.getAnswer();
            }
        }

        return null; // 未找到合适缓存
    }

    public void cacheAnswer(String question, String answer) {
        // 生成向量并存储
        Embedding embedding = embeddingModel.embed(question).content();
        CachedAnswer cached = new CachedAnswer(question, answer);
        vectorStore.add(embedding, cached);
    }
}

3) 分层缓存

@Service
public class TieredCacheService {

    private final Cache<String, String> localCache;
    private final RedisTemplate<String, String> redisCache;

    public TieredCacheService() {
        this.localCache = Caffeine.newBuilder()
            .maximumSize(1_000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .build();
    }

    public String get(String key) {
        // L1: 本地缓存
        String value = localCache.getIfPresent(key);
        if (value != null) {
            return value;
        }

        // L2: Redis缓存
        value = redisCache.opsForValue().get(key);
        if (value != null) {
            localCache.put(key, value);
            return value;
        }

        return null;
    }

    public void put(String key, String value) {
        // 写入L1和L2
        localCache.put(key, value);
        redisCache.opsForValue().set(key, value, 1, TimeUnit.HOURS);
    }
}

💡 缓存失效策略

  • 时间失效：设置TTL（Time To Live）
  • LRU：最少使用项优先淘汰
  • LFU：最不常使用项优先淘汰
  • 手动清除：提供主动清除接口
  • 智能失效：根据AI模型更新调整

✅ 最佳实践

  1. 合理设置TTL：平衡新鲜度和性能
  2. 监控缓存命中率：优化缓存策略
  3. 处理缓存穿透：对空结果也缓存
  4. 防止缓存雪崩：设置随机TTL
  5. 注意一致性：数据更新时清除缓存

5. Streaming Response（流式响应）

🎯 掌握程度：必须掌握

什么是流式响应？

通俗解释：
就像实时字幕逐字显示，流式响应让AI的回答逐步呈现，而不是等待完整答案。

技术定义：
服务器端生成响应后立即发送，客户端逐块接收和显示，实现实时交互体验。

为什么使用流式响应？

流式响应的优势：
✅ 用户体验：减少等待时间，即时反馈
✅ 感知性能：10秒响应比10秒等待感觉更快
✅ 中断能力：用户可以随时停止生成
✅ 资源利用：边生成边处理，提高效率
✅ 成本透明：实时看到Token消耗

实现方式

1) Server-Sent Events (SSE)

@RestController

            .doOnError(error -> log.error("Stream error", error));
    }
}

    private StreamingChatModel chatModel;

    @Override
    protected void handleTextMessage(WebSocketSession session,
                                   TextMessage message) throws Exception {
        String userMessage = message.getPayload();

        chatModel.stream(userMessage)
            .doOnNext(response -> {
                try {
                    String chunk = response.getResult().getOutput().getContent();
                    session.sendMessage(new TextMessage(chunk));
                } catch (IOException e) {
                    log.error("Failed to send message", e);
                }
            })
            .doOnComplete(() -> {
                try {
                    session.sendMessage(new TextMessage("[DONE]"));
                } catch (IOException e) {
                    log.error("Failed to send completion", e);
                }
            })
            .subscribe();
    }
}

        if (text) {

        }
    }

@Service
public class StreamProcessor {

    public Flux<String> processStream(Flux<String> stream) {
        return stream
            // 过滤空消息
            .filter(chunk -> !chunk.isEmpty())

            // 处理特殊标记
            .map(chunk -> chunk.replace("[DONE]", ""))

            // 限流：控制输出速度
            .delayElements(Duration.ofMillis(50))

            // 错误处理
            .onErrorResume(error -> {
                log.error("Stream error", error);
                return Flux.just("Error: " + error.getMessage());
            })

            // 超时处理
            .timeout(Duration.ofSeconds(30), Flux.just("Timeout"))

            // 完成通知
            .doOnComplete(() -> log.info("Stream processing completed"))

            // 错误日志
            .doOnError(error -> log.error("Stream processing failed", error));
    }
}

5) Token计数

@Service
public class TokenCounter {

    private static final Pattern TOKEN_PATTERN =
        Pattern.compile("\\S+"); // 简单Token估算

    public int countTokens(String text) {
        if (text == null || text.isEmpty()) {
            return 0;
        }

        Matcher matcher = TOKEN_PATTERN.matcher(text);
        int count = 0;
        while (matcher.find()) {
            count++;
        }
        return count;
    }

    public Flux<TokenStats> countStreaming(Flux<String> stream) {
        AtomicInteger totalTokens = new AtomicInteger(0);
        AtomicReference<String> fullText = new AtomicReference<>("");

        return stream
            .doOnNext(chunk -> {
                fullText.set(fullText.get() + chunk);
                int tokens = countTokens(chunk);
                totalTokens.addAndGet(tokens);
            })
            .map(chunk -> new TokenStats(
                chunk,
                countTokens(chunk),
                totalTokens.get()
            ));
    }
}

@Data
@AllArgsConstructor
class TokenStats {
    private String chunk;
    private int chunkTokens;
    private int totalTokens;
}

6. Orchestration（编排）

🎯 掌握程度：必须掌握

什么是编排？

通俗解释：
就像指挥家协调乐队演奏，AI编排协调多个AI组件协同工作，完成复杂任务。

技术定义：
协调和控制多个AI服务、工具和流程的组合，实现复杂的AI应用逻辑。

编排模式

模式类型 描述 适用场景
顺序编排 按顺序执行步骤 多阶段处理流程
并行编排 同时执行多个任务 独立任务处理
条件编排 根据条件选择路径 动态决策流程
循环编排 重复执行直到满足条件 迭代优化

1) 顺序编排

@Service
public class SequentialOrchestrator {

    @Autowired
    private EmbeddingModel embeddingModel;

    @Autowired
    private VectorStore vectorStore;

    @Autowired
    private ChatModel chatModel;

    public String processQuery(String query) {
        // 步骤1: 向量化查询
        Embedding queryEmbedding = embeddingModel.embed(query).content();

        // 步骤2: 检索相关文档
        List<Document> docs = vectorStore.findRelevant(queryEmbedding, 5);

        // 步骤3: 构建上下文
        String context = docs.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n\n"));

        // 步骤4: 生成回答
        String prompt = String.format("""
            基于以下上下文回答问题：

            上下文：
            %s

            问题：%s
            """, context, query);

        return chatModel.generate(prompt);
    }
}

2) 并行编排

@Service
public class ParallelOrchestrator {

    @Autowired
    private ChatModel chatModel;

    @Autowired
    private EmbeddingModel embeddingModel;

    @Autowired
    private SentimentAnalyzer sentimentAnalyzer;

    public Map<String, Object> analyzeContent(String content) {
        // 并行执行多个分析任务
        CompletableFuture<String> summaryTask = CompletableFuture.supplyAsync(
            () -> summarize(content)
        );

        CompletableFuture<String> sentimentTask = CompletableFuture.supplyAsync(
            () -> analyzeSentiment(content)
        );

        CompletableFuture<List<String>> keywordsTask = CompletableFuture.supplyAsync(
            () -> extractKeywords(content)
        );

        // 等待所有任务完成
        CompletableFuture.allOf(summaryTask, sentimentTask, keywordsTask).join();

        // 组合结果
        Map<String, Object> result = new HashMap<>();
        try {
            result.put("summary", summaryTask.get());
            result.put("sentiment", sentimentTask.get());
            result.put("keywords", keywordsTask.get());
        } catch (Exception e) {
            throw new RuntimeException("Analysis failed", e);
        }

        return result;
    }
}

3) 条件编排

@Service
public class ConditionalOrchestrator {

    @Autowired
    private IntentClassifier intentClassifier;

    @Autowired
    private ChatService chatService;

    @Autowired
    private SearchService searchService;

    @Autowired
    private TransactionService transactionService;

    public String handleRequest(String request) {
        // 步骤1: 分类意图
        Intent intent = intentClassifier.classify(request);

        // 步骤2: 根据意图路由
        return switch (intent.getType()) {
            case "CHAT" -> handleChat(request);
            case "SEARCH" -> handleSearch(request);
            case "TRANSACTION" -> handleTransaction(request);
            default -> handleUnknown(request);
        };
    }

    private String handleChat(String request) {
        return chatService.chat(request);
    }

    private String handleSearch(String request) {
        return searchService.search(request);
    }

    private String handleTransaction(String request) {
        return transactionService.execute(request);
    }

    private String handleUnknown(String request) {
        return "抱歉，我不理解您的请求。";
    }
}

@Service
public class LoopOrchestrator {

    @Autowired
    private ChatModel chatModel;

    @Autowired
    private QualityEvaluator qualityEvaluator;

    public String generateWithQualityCheck(String prompt) {
        String answer = chatModel.generate(prompt);
        int maxIterations = 3;
        int iteration = 0;

        while (iteration < maxIterations) {
            // 评估质量
            QualityScore score = qualityEvaluator.evaluate(prompt, answer);

            if (score.isAcceptable()) {
                return answer;
            }

            // 改进
            String improvedPrompt = String.format("""
                原始问题：%s

                之前的回答：%s

                评分：%d/100

                请改进回答，特别是：%s
                """, prompt, answer, score.getScore(), score.getFeedback());

            answer = chatModel.generate(improvedPrompt);
            iteration++;
        }

        return answer; // 返回最后一次结果
    }
}

5) 工作流编排

@Service
public class WorkflowOrchestrator {

    public String executeWorkflow(String input) {
        Workflow workflow = Workflow.builder()
            .step("validate", this::validate)
            .step("enrich", this::enrich)
            .step("process", this::process)
            .step("format", this::format)
            .build();

        return workflow.execute(input);
    }

    private String validate(String input) {
        if (input == null || input.trim().isEmpty()) {
            throw new IllegalArgumentException("Input cannot be empty");
        }
        return input;
    }

    private String enrich(String input) {
        // 添加上下文信息
        return "Context: " + getSystemContext() + "\n" + input;
    }

    private String process(String input) {
        // AI处理
        return chatModel.generate(input);
    }

    private String format(String input) {
        // 格式化输出
        return formatOutput(input);
    }
}

class Workflow {
    private final List<Step> steps;

    public String execute(String input) {
        String current = input;
        for (Step step : steps) {
            current = step.execute(current);
        }
        return current;
    }
}

6) Agent编排

@Service
public class AgentOrchestrator {

    private final Map<String, Agent> agents;

    public AgentOrchestrator() {
        this.agents = Map.of(
            "RESEARCHER", new ResearchAgent(),
            "WRITER", new WriterAgent(),
            "REVIEWER", new ReviewerAgent()
        );
    }

    public String executeWithAgents(String task) {
        // 步骤1: 研究Agent收集信息
        Agent researcher = agents.get("RESEARCHER");
        String researchResult = researcher.execute(task);

        // 步骤2: 写作Agent创建内容
        Agent writer = agents.get("WRITER");
        String draft = writer.execute(
            "基于以下研究撰写文章：\n" + researchResult
        );

        // 步骤3: 审核Agent评审
        Agent reviewer = agents.get("REVIEWER");
        String review = reviewer.execute(
            "评审以下文章：\n" + draft
        );

        // 步骤4: 根据评审修改
        if (review.contains("需要修改")) {
            draft = writer.execute(
                "根据评审修改文章：\n" + draft + "\n\n评审：" + review
            );
        }

        return draft;
    }
}
