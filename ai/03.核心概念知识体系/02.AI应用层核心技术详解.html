1. RAG（检索增强生成）

🎯 掌握程度：必须掌握

什么是RAG？

通俗解释：
RAG就像"开卷考试"。LLM是考生，知识库是教科书。考生回答问题时，可以翻阅教科书找到相关内容，然后基于这些内容回答问题。

技术定义：
Retrieval-Augmented Generation，通过检索外部知识库，增强LLM生成答案的准确性和可靠性。

为什么需要RAG？

问题1：LLM的知识有局限

LLM的问题：
❌ 训练数据有截止日期
   问："2025年最新的Java版本是什么？"
   答：不知道（训练数据只到2023年）

❌ 不懂私有知识
   问："公司的报销流程是什么？"
   答：不知道

❌ 容易"幻觉"
   问："我们公司有客户支持热线吗？"
   答：可能编造一个错误号码

RAG的解决方案

RAG的优势：
✅ 实时知识
   检索最新文档，获取最新信息

✅ 私有知识
   检索公司内部文档，获取私有知识

✅ 可靠答案
   基于真实文档，减少幻觉

✅ 可追溯
   标注答案来源，方便验证

💡 RAG vs Fine-tuning（微调）

RAG适用场景：

  • 需要频繁更新知识（如新闻、产品信息）
  • 需要引用来源（如法律、医疗文档）
  • 数据量不大但质量高
  • 避免模型幻觉

Fine-tuning适用场景：

  • 需要学习特定风格或格式
  • 数据量很大（百万级样本）
  • 需要降低推理成本

2. 相似度计算方法

余弦相似度（Cosine Similarity）:
similarity = cos(θ) = (A·B) / (||A|| × ||B||)
范围：[-1, 1]，越接近1越相似

欧氏距离（Euclidean Distance）:
distance = √Σ(ai - bi)²
范围：[0, ∞]，越小越相似

点积（Dot Product）:
similarity = A·B = Σ(ai × bi)
适合归一化后的向量

3. 检索优化技巧

✅ 混合检索（Hybrid Search）

结合向量检索和关键词检索（BM25），提升准确率

// LangChain4j RAG示例（简化版）
public class RagExample {
    public static void main(String[] args) {
        // 1. 加载文档
        DocumentLoader loader = new FileSystemDocumentLoader();
        List<Document> documents = loader.load("./docs");

        // 2. 文本分块
        TextSplitter splitter = new RecursiveCharacterTextSplitter(
            500,  // chunk size
            50    // overlap
        );
        List<TextSegment> segments = splitter.splitAll(documents);

        // 3. 向量化并存储
        EmbeddingModel embeddingModel = OpenAiEmbeddingModel.builder()
            .apiKey("your-api-key")
            .build();

        InMemoryEmbeddingStore<TextSegment> store = new InMemoryEmbeddingStore<>();
        for (TextSegment segment : segments) {
            Embedding embedding = embeddingModel.embed(segment).content();
            store.add(embedding, segment);
        }

        // 4. 检索
        String query = "什么是RAG？";
        Embedding queryEmbedding = embeddingModel.embed(query).content();
        List<EmbeddingMatch<TextSegment>> matches = store.findRelevant(queryEmbedding, 3);

        // 5. 构建Prompt并生成
        String context = matches.stream()
            .map(match -> match.embedded().text())
            .collect(Collectors.joining("\n"));

        String prompt = String.format("""
            基于以下信息回答问题：

            %s

            问题：%s
            """, context, query);

        ChatLanguageModel model = OpenAiChatModel.builder()
            .apiKey("your-api-key")
            .build();

        String answer = model.generate(prompt);
        System.out.println(answer);
    }
}

2. Vector Database（向量数据库）

🎯 掌握程度：必须掌握

什么是向量数据库？

通俗解释：
传统数据库按"完全匹配"查找（如：WHERE name = '张三'），向量数据库按"相似度"查找（如：找跟"苹果"意思相近的词）。

技术定义：
专门用于存储、索引和查询高维向量的数据库，支持高效的近似最近邻（ANN）搜索。

为什么需要向量数据库？

传统数据库的问题：
❌ 不支持语义搜索
   搜索"智能手机"，找不到"手机"
❌ 不支持相似度计算
   无法判断两个文档的相似程度
❌ 性能差
   高维向量计算非常慢

向量数据库的优势：
✅ 语义搜索
   理解词义、语境
✅ 高效检索
   索引技术加速查询（ANN算法）
✅ 大规模存储
   支持十亿级向量
✅ 实时更新
   支持增删改查

向量数据库核心概念

1. 向量（Embedding）

// 文本转向量示例
"苹果" → [0.1, -0.3, 0.8, ..., 0.5]  // 1536维（OpenAI）
"水果" → [0.2, -0.1, 0.9, ..., 0.4]

// 向量维度：通常在256-4096之间
// 数值范围：通常归一化到[-1, 1]或[0, 1]

2. 距离度量（Distance Metrics）

常用距离计算方法：

1. 欧氏距离（L2）
   distance = √Σ(xi - yi)²
   适用：图像相似度

2. 余弦相似度（Cosine）
   similarity = Σ(xi × yi) / (√Σxi² × √Σyi²)
   适用：文本相似度（最常用）

3. 内积（IP, Inner Product）
   similarity = Σ(xi × yi)
   适用：归一化后的向量，计算最快

4. 汉明距离（Hamming）
   统计不同位数的个数
   适用：二值向量

3. 索引算法（Index Algorithms）

💡 ANN（Approximate Nearest Neighbor）

精确搜索太慢，ANN用"近似"换"速度"，牺牲一点点精度换取100倍性能提升。

算法 原理 优点 缺点
FLAT 暴力遍历所有向量 100%召回率 慢，不适合大数据
IVF 聚类分区，只搜相关区 平衡速度和精度 需要调参nprobe
HNSW 图结构，多层导航 最快，精度高 内存占用大
ANNOY 随机森林分割 内存友好 构建索引慢
PQ 向量压缩（量化） 省内存，速度快 精度下降

主流向量数据库对比

数据库 特点 适用场景 Java生态支持
Milvus 开源，功能强大，性能高 生产环境首选 ⭐⭐⭐⭐⭐ SDK完善

Java操作Milvus

// Maven依赖
<dependency>
    <groupId>io.milvus</groupId>
    <artifactId>milvus-sdk-java</artifactId>
    <version>2.3.4</version>
</dependency>

// Java代码示例
public class MilvusExample {
    public static void main(String[] args) {
        // 1. 连接Milvus
        MilvusServiceClient client = new MilvusServiceClient(
            ConnectParam.newBuilder()
                .withHost("localhost")
                .withPort(19530)
                .build()
        );

        // 2. 创建Collection
        String collectionName = "documents";

        // 定义Schema
        FieldType fieldType1 = FieldType.newBuilder()
            .withName("id")
            .withDataType(DataType.Int64)
            .withPrimaryKey(true)
            .withAutoID(true)
            .build();

        FieldType fieldType2 = FieldType.newBuilder()
            .withName("embedding")
            .withDataType(DataType.FloatVector)
            .withDimension(1536)  // OpenAI dimension
            .build();

        CreateCollectionParam createParam = CreateCollectionParam.newBuilder()
            .withCollectionName(collectionName)
            .withFieldType(fieldType1, fieldType2)
            .build();

        R<RpcStatus> response = client.createCollection(createParam);

        // 3. 插入数据
        List<InsertParam.Field> fields = new ArrayList<>();

        // 准备向量数据
        List<List<Float>> vectors = new ArrayList<>();
        vectors.add(Arrays.asList(0.1f, 0.2f, ...));  // 1536维
        vectors.add(Arrays.asList(0.3f, 0.4f, ...));

        fields.add(new InsertParam.Field("embedding", vectors));

        InsertParam insertParam = InsertParam.newBuilder()
            .withCollectionName(collectionName)
            .withFields(fields)
            .build();

        client.insert(insertParam);

        // 4. 创建索引
        CreateIndexParam indexParam = CreateIndexParam.newBuilder()
            .withCollectionName(collectionName)
            .withFieldName("embedding")
            .withIndexType(IndexType.HNSW)  // 使用HNSW算法
            .withMetricType(MetricType.COSINE)
            .withExtraParam("{\"M\":16,\"efConstruction\":64}")
            .build();

        client.createIndex(indexParam);

        // 5. 加载Collection
        client.loadCollection(LoadCollectionParam.newBuilder()
            .withCollectionName(collectionName)
            .build());

        // 6. 向量搜索
        List<List<Float>> queryVectors = new ArrayList<>();
        queryVectors.add(Arrays.asList(0.15f, 0.25f, ...));  // 查询向量

        SearchParam searchParam = SearchParam.newBuilder()
            .withCollectionName(collectionName)
            .withVectorFieldName("embedding")
            .withVectors(queryVectors)
            .withTopK(10)  // 返回最相似的10个结果
            .withMetricType(MetricType.COSINE)
            .build();

        R<SearchResult> searchResponse = client.search(searchParam);

        // 处理搜索结果
        for (SearchResult.QueryResp queryResp : searchResponse.getData().getResults()) {
            for (Object id : queryResp.getIds().getIntId().getDataList()) {
                System.out.println("找到相似文档ID: " + id);
            }
        }
    }
}

向量数据库实战技巧

✅ 性能优化最佳实践

  1. 选择合适索引：
    • 小数据（<100万）：FLAT或IVF_FLAT
    • 中等数据（100万-1000万）：IVF_SQ8或HNSW
    • 大数据（>1000万）：HNSW + PQ
  2. 调整索引参数：
    • nlist（聚类数）：数据量的平方根（如100万数据→nlist=1024）
    • nprobe（搜索聚类数）：越大越准但越慢（推荐10-50）
  3. 使用过滤：
    • 先过滤后搜索（减少搜索范围）
    • 利用标量字段（如category、date）
  4. 批量操作：
    • 批量插入（每次1000-10000条）
    • 批量查询

⚠️ 常见问题

问题 原因 解决方法
搜索结果不相关 Embedding模型不合适 换更好的模型（如text-embedding-3-large）
查询很慢 没建索引或索引不当 创建HNSW索引，调整参数
内存占用太高 向量太多或维度太高 使用PQ压缩，降维（PCA）
精度不够 ANN算法牺牲了精度 增加nprobe，或用FLAT精确搜索

向量数据库 vs 传统数据库

场景对比：

1. 电商商品搜索
   传统数据库：按名称、ID精确搜索
   向量数据库：按图片、描述语义搜索

2. 用户推荐
   传统数据库：基于规则、标签匹配
   向量数据库：基于用户行为向量相似度

3. 文档问答
   传统数据库：全文关键词搜索（LIKE）
   向量数据库：语义理解，找到相关段落

3. Agent（智能体）

🎯 掌握程度：必须掌握

什么是Agent？

通俗解释：
LLM是"大脑"，Agent是"有手有脚的完整人"。

  • LLM：只能思考、回答问题
  • Agent：能思考+能行动（搜索网络、操作API、读取文件等）

技术定义：
AI Agent = LLM + Planning（规划）+ Memory（记忆）+ Tools（工具）

为什么需要Agent？

LLM的局限：
❌ 只能"说"，不能"做"
   用户："帮我查明天天气"
   LLM："对不起，我无法访问实时数据"

❌ 无法处理复杂任务
   用户："帮我规划一次日本旅行"
   LLM：需要分步骤（查机票、订酒店、查攻略...）

❌ 没有"记忆"
   对话结束就忘记上下文

Agent的能力：
✅ 自主规划
   复杂任务拆解成子任务

✅ 使用工具
   调用搜索、计算器、API等

✅ 记忆管理
   记住历史对话和结果

✅ 自主决策
   根据结果调整下一步行动

Agent关键技术

1. Planning（规划）

🔧 任务规划方法

ReAct框架（最常用）：

Thought（思考）：我需要做什么
Action（行动）：执行某个工具
Observation（观察）：得到什么结果
...循环...
Final Answer：最终答案

示例：
Thought：用户想知道北京的天气
Action：调用weather_api(city="北京")
Observation：得到"北京今天晴天，25度"
Thought：已经获取到天气信息

4. Function Calling（函数调用）

🎯 掌握程度：必须掌握

什么是Function Calling？

通俗解释：
让LLM能够"调用函数"。不是LLM自己执行代码，而是LLM告诉你"应该调用哪个函数，传什么参数"，然后你的代码去执行。

技术定义：
OpenAI等模型提供的功能，让LLM根据用户意图，输出结构化的函数调用请求（函数名+参数）。

为什么需要Function Calling？

传统LLM的局限：
❌ 只能返回文本
   用户："北京今天天气"
   LLM："我无法获取实时数据"

❌ 无法精确提取参数
   用户："帮我订明天北京到上海的机票"
   LLM：需要你自己解析"明天"、"北京"、"上海"

Function Calling的能力：
✅ 结构化输出
   LLM返回：{"function": "book_flight", "args": {...}}

✅ 参数提取
   自动从文本中提取结构化参数

✅ 函数选择
   根据用户意图选择合适的函数

✅ 多轮对话
   缺少参数时主动询问

Function Calling工作流程

┌─────────────────────────────────────────────────┐
│         Function Calling 完整流程                │
└─────────────────────────────────────────────────┘

1. 定义函数（告诉LLM有哪些函数可用）

   function getWeather(city: string, date: string) {
       // 调用天气API
   }

2. 用户提问

   用户："北京明天天气怎么样？"

3. LLM分析意图并返回函数调用

   LLM返回：
   {
     "function": "getWeather",
     "arguments": {
       "city": "北京",
       "date": "2025-03-10"
     }
   }

4. 代码执行函数

   getWeather("北京", "2025-03-10")
   → 返回："晴天，15-25度"

5. 把结果传回LLM

   把函数返回值告诉LLM

6. LLM生成最终答案

   LLM："北京明天晴天，气温15到25度"

使用OpenAI API

// Maven依赖
<dependency>
    <groupId>com.squareup.retrofit2</groupId>
    <artifactId>retrofit</artifactId>
    <version>2.9.0</version>
</dependency>

// 示例代码
public class FunctionCallingExample {
    public static void main(String[] args) {
        OpenAiApi service = new OpenAiService("your-api-key");

        // 1. 定义可用的函数
        List<ChatFunction> functions = List.of(
            ChatFunction.builder()
                .name("getWeather")
                .description("获取指定城市的天气信息")
                .parameters(ChatFunctionParameters.builder()
                    .type("object")
                    .properties(Map.of(
                        "city", Map.of(
                            "type", "string",
                            "description", "城市名称"
                        ),
                        "date", Map.of(
                            "type", "string",
                            "description", "日期（格式：YYYY-MM-DD）"
                        )
                    ))
                    .required(List.of("city", "date"))
                    .build())
                .build()
        );

        // 2. 发送请求（带上函数定义）
        ChatMessage userMessage = new ChatMessage(
            ChatMessageRole.USER.value(),
            "北京明天天气怎么样？"
        );

        ChatRequest request = ChatRequest.builder()
            .model("gpt-4")
            .messages(List.of(userMessage))
            .functions(functions)
            .functionCall("auto")  // 让模型自动决定是否调用函数
            .build();

        ChatResponse response = service.createChatCompletion(request);

        // 3. 检查LLM是否想调用函数
        if (response.getChoices().get(0).getMessage().getFunctionCall() != null) {
            ChatFunctionCall functionCall = response.getChoices()
                .get(0)
                .getMessage()
                .getFunctionCall();

            String functionName = functionCall.getName();
            String arguments = functionCall.getArguments();

            // 解析参数
            ObjectMapper mapper = new ObjectMapper();
            Map<String, Object> args = mapper.readValue(
                arguments,
                new TypeReference<Map<String, Object>>() {}
            );

            // 4. 执行函数
            String result = executeFunction(functionName, args);

            // 5. 把结果返回给LLM
            ChatMessage functionMessage = new ChatMessage(
                ChatMessageRole.FUNCTION.value(),
                result
            );
            functionMessage.setName(functionName);

            ChatRequest followUpRequest = ChatRequest.builder()
                .model("gpt-4")
                .messages(List.of(
                    userMessage,
                    response.getChoices().get(0).getMessage(),
                    functionMessage
                ))
                .build();

            ChatResponse finalResponse = service
                .createChatCompletion(followUpRequest);

            System.out.println(finalResponse.getChoices()
                .get(0)
                .getMessage()
                .getContent());
        }
    }

    private static String executeFunction(
        String functionName,
        Map<String, Object> args
    ) {
        if (functionName.equals("getWeather")) {
            String city = (String) args.get("city");
            String date = (String) args.get("date");
            // 调用真实的天气API
            return String.format(
                "%s %s 天气：晴天，气温15-25度",
                city, date
            );
        }
        return "Unknown function";
    }
}

5. Chain of Thought（思维链）

🎯 掌握程度：必须掌握

什么是Chain of Thought？

通俗解释：
让LLM"展示思考过程"。不是直接给答案，而是像数学题解题一样，一步步推理，最后得出答案。

技术定义：
一种Prompt技巧，引导模型生成中间推理步骤，提升复杂问题的解决能力。

为什么需要CoT？

传统LLM的问题：
❌ 直接回答，容易出错
   问："农夫有15只羊，除了6只都走了，还剩几只？"
   直接回答：9只（错误！）

❌ 复杂推理能力弱
   多步骤问题容易跳步

❌ 无法验证推理过程
   不知道答案怎么来的

CoT的优势：
✅ 逐步推理
   "先算总数15只，走了15-6=9只，还剩6只"

✅ 提高准确率
   复杂问题准确率提升30%+

✅ 可解释性
   能看到推理过程，便于调试

CoT核心方法

1. Zero-Shot CoT（零样本思维链）

方法：在Prompt中加一句"让我们一步步思考"

Prompt示例：
"""
问题：农夫有15只羊，除了6只都走了，还剩几只？
让我们一步步思考。
"""

模型输出：
1. 农夫总共有15只羊
2. "除了6只都走了"意思是只有6只没走
3. 所以还剩6只羊
4. 答案：6只

✅ 简单有效，无需示例

2. Few-Shot CoT（少样本思维链）

方法：提供几个带推理过程的示例

Prompt示例：
"""
Q：罗杰有5个网球，他又买了2罐网球，每罐3个。
   现在他有多少个网球？
A：罗杰一开始有5个球
   2罐网球 × 每罐3个 = 6个
   5 + 6 = 11个
   答案：11个

Q：食堂有23个苹果，如果他们用20个做午餐，
   又买了6个，现在有多少个？
A：让我们一步步思考。
"""

模型会模仿示例的推理风格。

3. Self-Consistency（自洽性）

💡 高级技巧

让模型生成多个推理路径，选择最一致的答案

// 伪代码
paths = []
for i in range(5):  // 生成5个推理路径
    path = llm.generate(question, cot_prompt)
    paths.append(path)

// 统计最频繁的最终答案
final_answer = most_common(paths)

优点：进一步提高准确率
缺点：成本增加（多次调用LLM）

什么是知识图谱？

通俗解释：
把现实世界的知识用"实体-关系-实体"的三元组表示，形成一个巨大的网络结构。

技术定义：
用图结构（节点和边）表示知识和关系的数据库技术。

知识图谱示例

三元组示例：
(苹果, 是一种, 水果)
(苹果, 产地, 中国)
(苹果, 颜色, 红色)
(水果, 富含, 维生素C)

图结构：
   苹果 ---是一种---> 水果
    |                    |
   产地                富含
    |                    |
   中国              维生素C
    |
   颜色
    |
   红色

知识图谱 vs 向量数据库

对比维度 知识图谱 向量数据库
数据结构 显式关系（三元组） 隐式语义（向量）
查询方式 图查询语言（Cypher、Gremlin） 向量相似度搜索
适用场景 复杂关系推理、路径查询 语义搜索、推荐系统
可解释性 强（可追溯关系路径） 弱（黑盒相似度）

Java操作Neo4j

// Maven依赖
<dependency>
    <groupId>org.neo4j.driver</groupId>
    <artifactId>neo4j-java-driver</artifactId>
    <version>4.4.9</version>
</dependency>

// 示例代码
public class KnowledgeGraphExample {
    public static void main(String[] args) {
        // 1. 连接Neo4j
        Driver driver = GraphDatabase.driver(
            "bolt://localhost:7687",
            AuthTokens.basic("neo4j", "password")
        );

        try (Session session = driver.session()) {
            // 2. 创建节点和关系
            session.run("CREATE (a:Person {name: '张三'})");
            session.run("CREATE (c:Company {name: '阿里巴巴'})");
            session.run("""
                MATCH (a:Person {name: '张三'})
                MATCH (c:Company {name: '阿里巴巴'})
                CREATE (a)-[:WORKS_AT]->(c)
                """);

            // 3. 查询
            Result result = session.run("""
                MATCH (p:Person {name: '张三'})-[:WORKS_AT]->(c:Company)
                RETURN p.name AS person, c.name AS company
                """);

            while (result.hasNext()) {
                Record record = result.next();
                System.out.printf("%s 在 %s 工作%n",
                    record.get("person"),
                    record.get("company")
                );
            }
        }
    }
}

知识图谱 + LLM

🔧 GraphRAG

结合知识图谱和RAG，提升检索质量

传统RAG问题：
- 检索到相关段落，但可能缺少上下文
- 无法理解实体间关系

GraphRAG优势：
1. 实体识别：从文本中提取实体（人名、公司名等）
2. 关系抽取：识别实体间关系
3. 知识图谱构建：存入图数据库
4. 图检索：找到相关实体+关系
5. 结合向量检索：混合搜索

示例：
用户："马云和阿里巴巴的关系？"
向量检索：可能找不到直接相关段落
知识图谱：马云 -[创始人]-> 阿里巴巴
