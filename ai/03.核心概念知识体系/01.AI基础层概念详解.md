## 1. LLM（大语言模型）

### 🎯 掌握程度：必须掌握

### 什么是LLM？

**通俗解释：**
LLM就像一个读了整个互联网的"超级图书馆管理员"。你问它任何问题，它都能基于海量知识给出回答。它不是搜索引擎，而是"理解+生成"内容的智能助手。

**技术定义：**
Large Language Model，基于深度学习的大规模语言模型，通过海量文本数据训练，能够理解和生成人类语言。

### 为什么重要？

- **AI时代的基石**：所有AI应用都建立在LLM之上
- **通用智能的雏形**：一个模型就能处理文本、代码、翻译等多种任务
- **开发门槛降低**：开发者不需要训练模型，直接调用API即可

### 能力与边界

#### ✅ LLM擅长的
- 文本理解和生成
- 代码编写和解释
- 语言翻译
- 逻辑推理（在一定范围内）
- 知识问答

#### ❌ LLM不擅长的
- 实时信息获取（训练数据截止日期之后的信息）
- 精确计算（容易算错）
- 长期记忆（需要外部存储）
- 理解私密或特定业务知识（需要RAG）

### Java开发者类比

```java
// 传统编程：你要明确告诉计算机每一步怎么做
public String generateGreeting(String name) {
    return "Hello, " + name + "!"; // 确定性输出
}

// LLM编程：你描述需求，AI自己决定怎么做
// 输入："生成一个友好的问候语，对象是张三"
// 输出："你好呀，张三！很高兴见到你！" // 创造性输出
```

## 2. Prompt Engineering（提示词工程）

### 🎯 掌握程度：必须掌握

### 什么是Prompt Engineering？

**通俗解释：**
Prompt就是"和AI说话的技巧"。就像和新人沟通要讲清楚需求一样，Prompt Engineering就是学会如何清晰、准确地告诉AI你想要什么。

**技术定义：**
通过精心设计输入提示词，引导LLM生成高质量、符合预期输出的技术和方法。

### 为什么重要？

```
好的Prompt = 好的输出
差的Prompt = 差的输出
```

- **决定AI应用质量**：Prompt是AI应用的"代码"
- **降低开发成本**：优化Prompt比训练模型便宜得多
- **快速迭代**：修改Prompt比改代码快

### Prompt编写框架

#### 🎯 CLEAR框架

```
C - Context（背景）：提供背景信息
L - Limit（约束）：设定限制条件
E - Example（示例）：给出参考例子
A - Action（动作）：明确要执行的动作
R - Result（结果）：描述期望的结果
```

### 实际示例

#### ❌ 差的Prompt
```
用户：写个用户登录功能
AI：生成了很多代码，但可能不符合你的需求
```

#### ✅ 好的Prompt
```
【背景】
我要为我的Spring Boot项目添加用户登录功能

【功能需求】
1. 支持邮箱+密码登录
2. 使用JWT进行认证
3. Token有效期为7天
4. 包含密码加密（BCrypt）

【技术栈】
- Spring Boot 2.7
- Spring Security
- MySQL数据库
- JPA

【约束条件】
- 密码长度至少8位
- 包含字母和数字
- 登录失败返回友好提示

【输出格式】
- 完整的Controller代码
- Service层实现
- Entity类定义
- Security配置

请按照这个需求生成代码。
```

### Prompt优化技巧

#### 技巧1：角色设定
```
你是一位有10年经验的Java架构师，精通Spring Boot...
```

#### 技巧2：分步引导
```
第一步：先告诉我你理解的需求是什么
第二步：给出技术方案
第三步：生成代码
```

#### 技巧3：示例驱动
```
示例：
输入："Hello"
输出："你好"

现在请翻译："Good morning"
```

#### 技巧4：思维链
```
请一步步思考：
1. 首先分析问题...
2. 然后考虑方案...
3. 最后给出结论...
```

## 3. Token（词元）

### 🎯 掌握程度：必须掌握

### 什么是Token？

**通俗解释：**
Token是AI"数钱"的单位。就像短信按条收费、流量按MB收费一样，LLM按Token收费。

**技术定义：**
LLM处理文本的最小单位，可以是一个词、一个词的一部分、或者一个字符。

### Token计算示例

#### 英文文本
```
文本: "Hello, world!"
Token数: 3 (Hello, / world, !)

文本: "Artificial Intelligence"
Token数: 2 (Artificial / Intelligence)
```

#### 中文文本
```
文本: "你好世界"
Token数: 约4-6个（取决于分词）

文本: "人工智能"
Token数: 约2-3个
```

### Token与计费

```
💰 成本计算公式：
总费用 = (输入Token数 × 输入单价 + 输出Token数 × 输出单价) / 1000

示例：
- 输入：1000 Tokens，价格 $0.001/1K tokens
- 输出：500 Tokens，价格 $0.002/1K tokens
- 总费用：(1000 × 0.001 + 500 × 0.002) / 1000 = $0.002
```

### Token与上下文限制

不同模型的Context Window（上下文窗口）不同：

| 模型 | Context Window | 说明 |
|------|---------------|------|
| GPT-3.5 | 4K/16K Tokens | 短对话 |
| GPT-4 | 8K/32K Tokens | 中长对话 |
| Claude 3 | 200K Tokens | 超长文档 |
| 通义千问 | 30K Tokens | 中等长度 |

### 优化Token使用

#### 策略1：精简输入
```
❌ 冗长：
"我想请你帮我写一个Java程序，这个程序的主要功能是实现一个用户登录的
功能，需要包含用户名和密码的验证..."

✅ 精简：
"用Java实现用户登录功能：邮箱+密码验证，JWT认证"
```

#### 策略2：使用引用
```
❌ 把整个文档都放进去
✅ 引用文档ID，让AI从知识库读取
```

#### 策略3：合理截断
```java
// 保留最重要的上下文
List<Message> history = getRecentMessages(sessionId, 10); // 只保留最近10条
```

### 实际示例

**场景：问答系统Token计算**

```bash
用户问题：Spring Boot中如何实现AOP？
         (约15 Tokens)

系统Prompt：你是一个Java技术专家...
           (约200 Tokens)

相关文档：从知识库检索到的AOP文档片段
          (约800 Tokens)

总计：约1015 Tokens（输入）
预计输出：约500 Tokens

费用估算：(1015 × 0.001 + 500 × 0.002) / 1000 = $0.002
```

### Java开发者注意事项

```java
// 传统编程：不关心字符数
String text = "很长的文本...";

// AI编程：需要关注Token数
public class TokenCounter {
    public int estimateTokens(String text) {
        // 粗略估算：英文约4字符=1token，中文约1.5字符=1token
        if (isChinese(text)) {
            return (int) (text.length() / 1.5);
        } else {
            return (int) (text.length() / 4);
        }
    }
}
```

## 4. Temperature（温度参数）

### 🎯 掌握程度：必须掌握

### 什么是Temperature？

**通俗解释：**
Temperature控制AI的"创造性程度"。温度低=保守准确，温度高=创新发散。

**技术定义：**
控制LLM输出随机性的参数，范围0-2，默认为1。

### Temperature效果对比

#### 低Temperature（0.1-0.3）
```
特点：
✅ 输出稳定、一致
✅ 适合需要准确答案的场景
❌ 缺乏创造性

适用场景：
- 代码生成
- 事实性问答
- 数据提取
- 文档总结

示例：
Prompt：用Java写一个冒泡排序
输出：每次都生成几乎相同的代码
```

#### 中等Temperature（0.7-1.0）
```
特点：
✅ 平衡创造性和准确性
✅ 通用场景首选

适用场景：
- 一般对话
- 内容创作
- 问题解答

示例：
Prompt：给我一个健康早餐的建议
输出：会在多个合理选项中变化
```

#### 高Temperature（1.0-2.0）
```
特点：
✅ 高度创造性
✅ 输出多样性强
❌ 可能不符合事实

适用场景：
- 创意写作
- 头脑风暴
- 艺术创作

示例：
Prompt：讲一个科幻故事
输出：每次都完全不同的故事
```

### 实际示例

#### 示例1：代码生成（低温度）

```bash

输出2：

    if (n <= 1) return false;
    for (int i = 2; i <= Math.sqrt(n); i++) {
        if (n % i == 0) return false;
    }
    return true;
}
// 几乎完全相同
```

#### 示例2：创意写作（高温度）

```bash

Prompt：写一个关于AI的简短故事

输出1：
2150年，第一个有自我意识的AI诞生了...
（科幻风格）

输出2：
小明是个程序员，他开发的AI助手帮他找到了女朋友...

输出3：
```

## 5. Context Window（上下文窗口）

### 🎯 掌握程度：必须掌握

### 什么是Context Window？

**通俗解释：**
Context Window是AI的"短期记忆容量"。就像人一次只能记住有限的信息一样，AI一次能处理的文本长度也是有限的。

**技术定义：**
LLM一次能处理的最大Token数量限制，包括输入和输出。

### 不同模型的Context Window

| 模型 | Context Window | 实际文本量 | 适用场景 |
|------|---------------|-----------|----------|
| **GPT-3.5-turbo** | 4K/16K | 约10-40页文档 | 短对话、简单任务 |
| **GPT-4** | 8K/32K | 约20-80页文档 | 中长文档、复杂任务 |
| **Claude 3 Opus** | 200K | 约500页文档 | 超长文档、书籍分析 |
| **Claude 3 Sonnet** | 200K | 约500页文档 | 长文档处理 |
| **通义千问-turbo** | 8K | 约20页文档 | 中文场景 |
| **通义千问-plus** | 32K | 约80页文档 | 复杂中文任务 |

### Context Window限制的影响

#### 问题1：超出限制怎么办？

```bash
场景：分析一个100页的PDF文档

❌ 错误做法：
直接把整个文档发给AI
结果：报错 "Context length exceeded"

✅ 正确做法：
1. 分块处理（Chunking）
2. 只提取关键部分
3. 使用RAG技术（后续详解）
```

#### 问题2：对话历史管理

```bash
场景：长对话系统

问题：
随着对话进行，历史消息越来越多，最终超出限制

解决方案：
1. 只保留最近N条消息
2. 总结历史对话
3. 分层次存储重要信息
```

### Context管理策略

#### 策略1：滑动窗口

```java
public class ConversationManager {
    private static final int MAX_CONTEXT = 4000; // Token限制

    public List<Message> getContext(String sessionId) {
        List<Message> history = getHistory(sessionId);

        // 从最新的消息开始，往前取，直到接近Token限制
        List<Message> context = new ArrayList<>();
        int totalTokens = 0;

        for (int i = history.size() - 1; i >= 0; i--) {
            Message msg = history.get(i);
            int tokens = estimateTokens(msg);

            if (totalTokens + tokens > MAX_CONTEXT) {
                break;
            }

            context.add(0, msg);
            totalTokens += tokens;
        }

        return context;
    }
}
```

#### 策略2：智能摘要

```java
// 当对话过长时，总结旧对话
public String summarizeOldMessages(List<Message> oldMessages) {
    String prompt = "请总结以下对话的要点：\n" +
                    oldMessages.toString();
    return llmService.summarize(prompt);
}
```

#### 策略3：分层存储

```
短期记忆（最近10条）：
├─ 完整消息内容
└─ 直接放入Context

中期记忆（最近100条）：
├─ 存储在数据库
└─ 按需检索

长期记忆（所有历史）：
├─ 存储在向量数据库
└─ 使用RAG检索
```

### 实际示例

#### 场景：智能客服系统

```java
@Service
public class CustomerServiceBot {

    private static final int MAX_TOKENS = 3000;
```

### Java开发者注意事项

```java
// 传统编程：不需要关心上下文长度
List<String> messages = getAllMessages(); // 可以无限长

// AI编程：必须管理上下文
public class ContextManager {

    // ✅ 好：估算Token数
    public boolean canFitInContext(String text, int limit) {
        int tokens = estimateTokens(text);
        return tokens < limit;
    }

    // ✅ 好：自动截断
    public String truncateToFit(String text, int maxTokens) {
        int currentTokens = estimateTokens(text);
        if (currentTokens <= maxTokens) {
            return text;
        }

        // 按比例截断
        double ratio = (double) maxTokens / currentTokens;
        int keepChars = (int) (text.length() * ratio);
        return text.substring(0, keepChars);
    }
}
```

### 最佳实践

```
✅ 推荐做法：
1. 提前规划Token预算
2. 实现Token计数工具
3. 设置安全阈值（如上限的90%）
4. 记录Token使用情况
5. 实现自动截断和总结

❌ 避免做法：
1. 忽略Token限制
2. 假设所有内容都能放下
3. 不做Token估算
4. 没有异常处理
```

## 6. Embedding（向量化）

### 🎯 掌握程度：必须掌握

### 什么是Embedding？

**通俗解释：**
Embedding就是把文字变成"数字坐标"，让计算机能理解文本的语义。就像地图上每个位置都有坐标(x,y)一样，Embedding给每个词、每句话一个"语义坐标"。

**技术定义：**
将文本、图像等高维数据映射到低维连续向量空间的技术，使得语义相似的内容在向量空间中距离相近。

### 为什么需要Embedding？

```
问题：计算机不理解"文字"的含义

传统方法：关键词匹配
搜索："苹果手机"
❌ 找不到："iPhone"（关键词不匹配）

Embedding方法：语义理解
搜索："苹果手机"
✅ 找到："iPhone"（语义相近）
```

### Embedding工作原理

#### 文本 → 向量

```
文本： "苹果"
向量：[0.12, -0.34, 0.56, ..., 0.78]
      （1536维的数字数组）

文本： "香蕉"
向量：[0.11, -0.33, 0.57, ..., 0.77]

文本： "手机"
向量：[0.65, -0.12, 0.34, ..., 0.92]

计算相似度：
相似度("苹果", "香蕉") = 0.98  # 都是水果，语义接近
相似度("苹果", "手机") = 0.25  # 语义不相关
```

### 向量相似度计算

```java
public class VectorUtils {

    // 计算余弦相似度
    public static double cosineSimilarity(float[] v1, float[] v2) {
        double dotProduct = 0.0;
        double norm1 = 0.0;
        double norm2 = 0.0;

        for (int i = 0; i < v1.length; i++) {
            dotProduct += v1[i] * v2[i];
            norm1 += v1[i] * v1[i];
            norm2 += v2[i] * v2[i];
        }

        return dotProduct / (Math.sqrt(norm1) * Math.sqrt(norm2));
    }
}

// 使用示例
float[] apple = embeddingService.embed("苹果");
float[] banana = embeddingService.embed("香蕉");

double similarity = VectorUtils.cosineSimilarity(apple, banana);
// 结果：约0.98（非常相似）
```

### Java实现示例

```java
@Service
public class EmbeddingService {

    private final OpenAIClient openAIClient;

    // 将文本转向量
    public float[] embed(String text) {
        EmbeddingRequest request = EmbeddingRequest.builder()
            .model("text-embedding-ada-002")
            .input(text)
            .build();

        EmbeddingResponse response = openAIClient.createEmbedding(request);
        return response.getData().get(0).getEmbedding();
    }

    // 批量转向量（更高效）
    public List<float[]> batchEmbed(List<String> texts) {
        EmbeddingRequest request = EmbeddingRequest.builder()
            .model("text-embedding-ada-002")
            .input(texts)
            .build();

        EmbeddingResponse response = openAIClient.createEmbedding(request);
        return response.getData().stream()
            .map(Embedding::getEmbedding)
            .collect(Collectors.toList());
    }

    // 找最相似的文本
    public String findMostSimilar(String query, List<String> candidates) {
        float[] queryVector = embed(query);
        List<float[]> candidateVectors = batchEmbed(candidates);

        double maxSimilarity = -1;
        int bestIndex = 0;

        for (int i = 0; i < candidates.size(); i++) {
            double similarity = cosineSimilarity(
                queryVector,
                candidateVectors.get(i)
            );
            if (similarity > maxSimilarity) {
                maxSimilarity = similarity;
                bestIndex = i;
            }
        }

        return candidates.get(bestIndex);
    }
}
```

## 7. 模型推理 vs 训练

### 📖 掌握程度：需要了解

### 基本概念

#### 模型训练（Training）

**通俗解释：**
训练就像"培养一个专家"。需要大量数据、大量计算资源、很长时间，让模型学会知识。

**特点：**
- 成本高昂（百万到千万美元）
- 需要大量数据（TB级别）
- 需要专业团队
- 时间周期长（数月到数年）

#### 模型推理（Inference）

**通俗解释：**
推理就像"使用专家"。模型已经训练好了，你只需要调用它，让它完成任务。

**特点：**
- 成本低廉（按Token计费）
- 无需大量数据
- 任何开发者都能用
- 实时响应（秒级）

### 为什么应用层开发者只需要关注推理？

```
训练层（研发层）：
├─ OpenAI、Anthropic、阿里云等公司负责
├─ 研发新模型
├─ 优化模型性能
└─ 需要AI专家、大量资源

应用层（我们）：
├─ 使用训练好的模型
├─ 开发AI应用
├─ 解决业务问题
└─ 需要业务理解、编程能力
```

### 类比说明

```
汽车制造：
├─ 制造引擎（训练）：
│   ├─ 需要工厂、工程师、大量资金
│   └─ 汽车厂商负责
│
└─ 开车（推理）：
    ├─ 只需要驾照、汽车
    └─ 任何人都能学会

AI开发：
├─ 训练模型（训练）：
│   ├─ 需要GPU集群、AI专家、大量数据
│   └─ AI厂商负责
│
└─ 调用模型（推理）：
    ├─ 只需要API、编程基础
    └─ 任何开发者都能用
```

### 何时需要训练？

**99%的应用场景不需要训练，只需：**

#### 方案1：Prompt Engineering（首选）
```
优化提示词，让模型更好完成任务
成本：几乎为0
效果：对大多数场景足够好
```

#### 方案2：RAG（检索增强）
```
给模型提供相关知识
成本：低
效果：解决知识不足问题
```

#### 方案3：Fine-tuning（微调）
```
仅在以下情况考虑：
✅ 有大量高质量数据（1000+样本）
✅ 需要特定领域知识
✅ 有专业团队和预算
❌ 绝大多数业务场景不需要
```

## 8. Transformer架构

### 📖 掌握程度：需要了解

### 什么是Transformer？

**通俗解释：**
Transformer是现代AI的"引擎"。就像内燃机是汽车的核心技术一样，Transformer是ChatGPT、Claude等所有现代LLM的基础架构。

**技术定义：**
一种基于注意力机制（Attention Mechanism）的深度学习架构，2017年由Google提出，彻底改变了NLP领域。

### 为什么重要？

```
AI发展时间线：
2017年之前：RNN、LSTM时代
├─ 处理序列数据效率低
├─ 无法并行计算
└─ 长距离记忆差

2017年：Transformer诞生
├─ 《Attention Is All You Need》论文
├─ 可以并行计算
├─ 长距离记忆能力强
└─ 成为现代AI基础

2018年后：LLM爆发
├─ GPT系列（基于Transformer）
├─ BERT（基于Transformer）
├─ Claude、通义千问等
└─ 都是基于Transformer
```

### 核心概念：注意力机制（Attention）

**通俗解释：**
注意力机制就像人阅读时会"重点关注"某些词。

**示例：**
```
句子："The animal didn't cross the street because it was too tired"

Question: "it" 指代什么？

传统RNN：可能搞混

Transformer注意力机制：
"it" ← 关注 → "animal" (相似度0.8)
"it" ← 关注 → "street" (相似度0.2)
结论："it" 指代 "animal"
```

### Transformer vs Java开发

```
Transformer架构：
├─ Encoder（编码器）：理解输入
├─ Decoder（解码器）：生成输出
└─ Self-Attention：关注重要信息

类比Java开发：
├─ 读取代码（编码）：理解需求
├─ 编写代码（解码）：实现功能
└─ 重点突出（注意力）：关注关键逻辑
```

### 为什么Java开发者只需了解？

```
应用层开发者：
├─ 使用训练好的模型
├─ 不需要理解架构细节
├─ 就像开车不需要懂发动机制造

了解Transformer的价值：
✅ 理解LLM能力来源
✅ 知道为什么需要大量计算
✅ 理解Token机制
✅ 建立AI认知框架

不需要深入：
❌ 具体数学公式
❌ 梯度反向传播
❌ 参数优化方法
```

### 学习建议

**了解即可：**
- Transformer是现代AI基础
- 基于注意力机制
- 支持并行计算
- 是所有LLM的基础

**不需要：**
- 深入数学原理
- 实现Transformer
- 训练Transformer模型
